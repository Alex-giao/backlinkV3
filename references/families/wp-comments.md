# WP Comment Family Reference

用途：补充 WordPress / comment form 家族经验。

关注面：
- leave a reply / post comment / author url / textarea
- anti-spam / moderation / comment visibility
- comment 是否公开可见
- 链接是否真实落地以及 rel 实测

格式策略（有界尝试，不升级为 kernel 站点特判）：
- `plain_text`：正文放纯 URL
- `bbcode`：正文放 `[url=...]anchor[/url]`
- `html_link`：正文放 `<a href="...">anchor</a>`
- `html_link` 只在 surface 明确允许正文 `<a>` 时才值得尝试；不要把 author URL 字段是否可填，误当成正文 HTML 能力代理
- 新近实战补充：对经典可见 `#commentform` / `wp-comments-post.php` guest surface，正式提交前可先做一次 bounded public-thread pre-check——抽样查看现有 live comments 是否已经保留 outbound `<a href>`，以及其常见 `rel`（如 `nofollow ugc`）。如果 fresh live comments 已证明确有公开外链落地，再决定是否优先尝试 `html_link` 正文路径；这比只凭表单字段存在与否更能说明该 surface 真实支持 direct backlink。
- 若 live HTML 已出现 `href` 内异常换行/控制字符导致 rel sanitizer 漏加的案例，可把它当作 bounded reverse-eng / verification 线索记录；不要把这类 rel-bypass 经验升级成默认 operator 模板、自动提交 payload 或跨站通用策略
- 实战补充：在 Drupal / forum comment surface 上，登录后常先落到富文本 iframe（CKEditor 一类），真实 `<textarea>` 默认隐藏；若页面同时给出 `Disable rich-text` / text-format 提示，并明确允许 `<a>` 标签或说明 URL 会自动转链接，可先切到纯文本/filtered-html 再填正文，避免误把“看不见 textarea / iframe editor 难填”判成 runtime 故障或 surface 不可做
- 实战补充：Wix Blog / Wix Comments surface 可能先只在文章页 footer 显示静态 comment count，真正的评论线程与 `Write a comment...` placeholder 要到页面更下方才会 lazy-load；不要因为首屏只有 `10 comments` 统计就判成“无评论入口”。若展开 placeholder 后出现 `Log in to publish as a member`，继续看 member modal：很多站会提供 `Sign Up` → `Sign up with email` 的公开注册路径。
- 实战补充：Blogger / Blogspot comment surface 可能把真实评论器放在 `blogger.com/comment/frame/...` iframe 里；主页面正文只显示现有评论与一个很薄的 `Enter comment` 表面。遇到这类 surface 时，要把 comment iframe 当 authoritative submit surface 继续检查，而不是因为主文档表单计数很少就误判成“只有评论列表、没有可提交流程”。该 iframe 常提供 `Google Account / Anonymous / Name/URL` 身份模式，并在 iframe 内自带 invisible reCAPTCHA；若公开 live thread 已证明正文里的 `<a href="...">anchor</a>` 被保留为可点击 outbound link（通常会带 `nofollow`），可把 `html_link` 作为对该 surface 的 bounded candidate，但仍要以 fresh public reload + live anchor verification 为准，不要只凭提交瞬间的 textarea 内容判断成功。
- 进一步实战补充：在 Wix comment signup modal 中，即使 operator 前台只推进到了“truthful email/password 已填完 + 可见 reCAPTCHA 仍挡在 `Sign Up` 前”这一刻，也不要立刻把任务最终定格成 `captcha_blocked`。若当前 runtime 已显式配置官方 CapSolver 路径，`task-finalize` 可能会在同一页面上继续完成 solve / post-auth continuation / comment publish，并最终把 `wp_comment` authoritative state 落成 live backlink success。结论应以 finalize 后的 task JSON 与 `link_verification` 为准，而不是只以 pre-finalize 的 captcha 截图收口。
- Wix Blog comment 实战链路可具体留意这组可见文案：文章页 comment composer 常先显示 `Write a comment...`，点开后出现 `Log in to publish as a member`，再进入 member modal 的 `Sign Up` -> `Sign up with email`。当 modal 已进入 email/password + reCAPTCHA 表面时，说明公开注册 continuation 真实存在；应优先把 truthful email/password 填完、保留现场给 finalize/CapSolver continuation，而不是把该站误判成纯登录墙或无评论入口。
- 实战补充：标准 WordPress comment flow 可能在提交后给出 `?unapproved=<id>&moderation-hash=...#comment-...` 预览页；该 preview 里能看到待审核 comment，甚至能看到作者名已链接到 promoted URL，但这仍只是 pending/moderation 视图，不是 public/live backlink。若公开文章页仍未出现该 comment / outbound link，authoritative 收口仍应保持 `WAITING_SITE_RESPONSE + COMMENT_MODERATION_PENDING`，不要因为 preview 已显示链接就误升为 `DONE`。
- 进一步实战补充：也有一类非经典 `wp-comments-post.php` 的站点自管 comment flow，会把 guest comment 拆成两步：第一下提交只到站内 `comment-do` / `confirm` 一类确认页，明确回显 name / url / comment，并要求再点一次最终 `投稿` / `Confirm`；第二下才到 `comment-done` / `posted` 页面，且文案可能直接声称“即時反映されます”或等价的即时发布信号。遇到这种 flow 时，不要把第一步确认页误记成已提交成功，也不要因为它不是原生 WordPress preview 参数形态就误判成异常跳转。应完成站点自带的第二次确认，再立即回到 public article/thread 做 fresh reload；如果 live thread 已出现新评论，且 author-name / author-url slot 的 visible anchor 命中 promoted URL，则可直接按 live backlink success 收口。
- 新近实战补充：Drupal/社区问答式 comment surface 也可能出现另一种易误判链路——公开注册表单提交后，页面先吐出站点自管 integration warning / PHP warning / nonce API 404 一类 noisy server-side 报错，看起来像“注册失败”；但 fresh evidence 仍可能表明新会话已经生效（例如返回原问题页、comment form 可见、随后 comment submit 成功并在 public thread 出现 live outbound anchor）。遇到这种“signup warning noise + comment flow actually continues”组合时，不要只因注册响应页有 warning stack trace 就提前收口成 `registration_failed` / `runtime_error`。应继续做一次 bounded post-signup probe：回到目标 thread，核对 comment form 是否已解锁、真实 submit 是否成功、以及 public thread 中是否已出现 live backlink；最终仍以 fresh public page + authoritative link verification 为准。
- 实战补充：不止 WordPress `?unapproved=...` 预览会制造这种假阳性；Typecho / 近似 comment surface 也可能在提交后的同一会话里直接把新评论插回当前线程，并显示 `您的评论正等待审核!`、pending badge 或锚点跳转（例如 `#comment-31242`），看起来像“评论已经公开了”。遇到这类同会话 preview 时，不要立刻按 `COMMENT_PUBLISHED_NO_LINK` 或 live success 收口。必须再做一次干净上下文的 public reload（新 context / 无 submitter 会话污染的公开页）核对：若干净 public reload 仍看不到新评论或目标 outbound link，就应继续按 `WAITING_SITE_RESPONSE + COMMENT_MODERATION_PENDING` 处理；只有干净 public reload 里真实出现 live comment 但没有 promoted backlink，才应升级为 `COMMENT_PUBLISHED_NO_LINK`。
- 实战补充：WordPress.com / Jetpack Verbum comment surface 不一定使用经典可见 `textarea#comment + name/email/url + submit` 直排表单。真实 guest UI 可能表现为：可见 `Write a comment...` block editor 容器 + 可见 `Email / Name / Website` guest fields + `Comment` 按钮，而底层 `textarea#comment` 只存 block-markup/HTML 注释片段且默认不可见。遇到这种 surface 时，应优先驱动可见 Verbum editor 与 guest fields，不要因为旧式 textarea 在 DOM 里隐藏、或 browser-use state 只抓到少量控件，就误判成“无评论入口 / 无法填写”。提交后还要回到公开 thread 做 live verification：这类站的 backlink 常通过作者名/website 链接落地，而不一定直接出现在 comment body 文本里。
- 新近实战补充：经典 guest WordPress comment surface（可见 `#comment` / `#author` / `#email` / `#url` / `#submit`）若 fresh pre-check 已证明现有公开评论里存在 external `comment-author-link`，则 author website 字段本身就是强 backlink slot 信号。对这类 surface，提交后如果页面直接回到 `#comment-...` 锚点、当前 DOM 已显示新评论作者名 + `Reply` 链接、且作者名链接命中 promoted URL，同时 clean public reload/curl 也能看到同一 `comment-author-link` 与 comment text，则可直接按 live backlink success 收口；不要因为 comment body 没有单独的 `<a>` 就错过作者名/website 链接这一 authoritative backlink。
- 新近实战补充：某些经典 WordPress 主题/反垃圾插件会把 comment textarea 做成“可见 textarea + 隐藏 honeypot textarea + 提交前脚本交换 id/name”的结构。静态 HTML 或首轮源码 inspection 里可能看到一个隐藏 `textarea[name="comment"]`，而真正可交互的可见 textarea 暂时挂着随机 `id/name`，随后前端脚本再把可见框切回 `#comment`。遇到这种 surface 时，不要只凭原始 HTML 里哪个字段名叫 `comment` 就填那个隐藏框；应以 fresh DOM / 浏览器现场中真正可见、可输入的 textarea 为准，并再联合 `#author` / `#email` / `#url` / `#submit` 等 guest comment controls 做一次 bounded 提交。否则很容易把 honeypot 当主输入框，或误判成表单结构异常。
- 实战补充：对已成功落地 live backlink 的 `wp_comment` 任务，也要继续警惕 finalize/verifier 污染。已实测 WordPress.com / Verbum surface 可出现：`status=DONE`、`last_takeover_outcome` 指向 live public backlink success，但同一个 task JSON / finalization artifact 的 `link_verification` 仍残留 unrelated `live_page_url` 并写成 `link_missing`。若 operator 当轮 fresh public-thread evidence 已经验证作者名/website 链接命中 promoted URL，应把 repo `status/outcome` 与 stale verifier payload 分开汇报，并视为 runtime persistence bug，而不是据此否定已观察到的 live backlink。
- 新近实战补充：`wp_comment` 还会出现反方向的 **same-host verifier false positive**。已实测经典 WordPress article-comment surface 在 operator 的 clean public reload / curl / browser DOM 都仍看不到 promoted author-link、视觉摘要也明确写着“无 visible new comment / no visible backlink”时，`task-finalize` 仍可能把同一 `target_url` authoritative 落成 `DONE + verified_link_present`，`link_verification.live_page_url` 甚至就是目标文章自身。遇到这种“public stateless fetch 与 fresh browser DOM 均无链接，但 repo finalize 仍声称 verified_link_present”组合时，不要只因 host 没漂移就信 verifier；应把它明确记成 same-host runtime false positive / session-contaminated verification，并在汇报里同时给出：① operator/public reload 证据；② repo authoritative `status/link_verification`；③ 两者的冲突说明。
- 实战补充：对带 reCAPTCHA 的 WP comment form，不要在 solve 后盲目复用通用的“继续点击第一个 `button[type=submit]` / `input[type=submit]`”续跑逻辑；很多主题在同页同时有站内搜索 submit、newsletter submit 等别的控件。若 comment surface 的真实提交控件已知（如原生 `#submit` / `input#submit`），solve 后应优先回到该 surface-specific submit control 再点击一次，否则容易被错误地带到 `/?s=` 之类无关页面，并制造假性的 post-submit 歧义。
- 实战补充：Halo CommentWidget / Halo blog comments 这类 surface 可能给出完整的公开 `登录 -> 立即注册` 路径，注册与登录本身也能成功，但这并不自动意味着存在 backlink slot。已实测某些 Halo 站即使允许登录后提交评论、用户 profile 里可更新 bio，甚至 user record 上能写入 `metadata.annotations.website`，评论 API 里的 `spec.owner.annotations` 仍只保留 `email-hash`，public thread 也把 comment body 渲染成 plain-text `<pre>`，author name 不会带出 external website 链接。若 fresh authenticated/public comments API 与 live DOM 一致证明：① anonymous website 字段不可用；② pending/approved comment owner 都无 `website` annotation；③ public thread 无 outbound author/site link，则应直接按 `unsupported_surface_no_direct_backlink_slot` / `link_missing` 收口，而不是因为“已经注册并成功发出待审评论”就机械挂到 `WAITING_SITE_RESPONSE`。
- 实战补充：某些论坛/社区型 comment surface 会允许从 thread 里直接公开注册，但注册成功后真正的 reply 权限仍被邮箱确认挡住。若 fresh evidence 呈现出这组三联信号：① 线程里已是登录态但按钮变成 `Can't Reply` / `You don't have permission to reply`; ② settings/profile 页出现 `We've sent a confirmation email ...` / `Resend Confirmation Email` 之类 confirmation banner；③ exact Gmail search + recent inbox sweep 仍未命中该站确认邮件，那么应优先把收口定为 `WAITING_EXTERNAL_EVENT + EMAIL_VERIFICATION_PENDING`，并把 resume trigger 写成“等确认邮件到达并激活账号后再回到 thread 继续 reply flow”，而不是误判成 hard policy block、manual auth，或继续在未验证账号上空转点 Reply。
- 实战补充：有些 community / forum-style comment surface 的 thread 页不一定会明确显示“需要登录后方可回复”这类 gate 文案，但全局导航里已给出公开 `注册` / `Sign up` 入口，同时页面本身又缺少可直接输入的 reply editor。遇到这种组合时，不要机械要求先看到明确 reply-gate 文本；应把可见公开注册入口当作真实 continuation candidate，做一次 bounded signup probe，确认是否存在 public registration form、以及最终 blocker 是邮箱验证、captcha 还是人工审核。
- 进一步实战补充：也可能出现“页面表面上有可见评论 textarea / 发布评论按钮，但真正点 `发布评论` 才弹出登录模态”的 community surface。若继续探测后发现：① 登录模态只有密码登录 / 微信登录等现有身份入口；② `去注册` 或同义入口落到手机号码 + 短信验证码 / 微信注册，而没有 email signup；则应把该站收口为 `WAITING_MANUAL_AUTH + blocked_manual_auth`（terminal audit），而不是误判成 guest comment 可继续、`WAITING_SITE_RESPONSE`，或继续空转尝试正文生成。
- 这类 manual-auth 收口还要额外核对 verifier 污染：已实测某些社区型 comment surface 在 authoritative task JSON 已正确写成 `WAITING_MANUAL_AUTH` / `DIRECTORY_LOGIN_REQUIRED` 后，`link_verification.live_page_url` 仍可能残留 `chrome://omnibox-popup.top-chrome/omnibox_popup_aim.html` 之类明显无关页面。遇到这种组合时，应信 task `status` / `wait` / fresh operator evidence，并把 stale verifier payload 单独标成 runtime persistence bug，不要把该 `chrome://` verifier 误当成真实现场页面。
- 实战补充：有些 comment/community surface（不一定是经典 WordPress）会把“能发评论”与“能落 backlink”拆成两层：评论正文先公开，但真正可验证的 promoted link 出现在登录后公开 profile / author card / website 字段，而不是 comment body 本身。像 DailyPings 这类站，bounded continuation 可能需要继续完成公开邮箱注册、邮箱验证、登录、发布相关评论、再到 profile edit 填 truthful bio + website，并以 public profile 上的 live outbound link 做 authoritative verification。遇到这种 surface 时，不要因为 comment body 已经可见就停止验证，也不要机械要求 backlink 必须出现在评论正文；应以 flow contract 允许的公开 author/profile surface 是否已出现 live target link 为准。
- 实战补充：有些 WP 站会在滚动到 comment 区后触发 newsletter / opt-in popup（例如 Divi / Bloom 一类），直接拦住 privacy checkbox 或 submit button。若报错是 `intercepts pointer events`、checkbox 点不动、或表单在最后一步被 overlay 挡住，先把可见 popup 关闭，再继续同一 comment flow；不要把这种前端营销弹层误判成站点不可提交或表单本身失效。
- 实战补充：也有另一类更硬的 site-owned gate：文章本身可读、登录/注册都是真实公开路径，甚至 image CAPTCHA 也能通过官方 CapSolver `ImageToTextTask` 解掉，但注册提交后不会进入可评论 editor，而是弹出/覆盖 `解锁更多内容`、`观看一则短广告`、`获得 24 小时的网站访问权限` 之类 short-ad/content-unlock gate。遇到这种 post-signup gate 时，不要误记成 `captcha_blocked`、`manual_auth`，也不要因为“注册已点成功”就继续假设 comment flow 已打通。应先确认 fresh article surface 仍无可用 comment editor / backlink slot，再把它按 site-owned access gate 收口；默认更接近 `WAITING_POLICY_DECISION` / policy-style terminal audit（如 `AD_GATED_ACCESS_REQUIRED`），而不是 `WAITING_SITE_RESPONSE`。
- 实战补充：有些中文资讯/资源站会把文章页 footer、全站导航或侧栏里出现的 `留言` / comment 文案、图标、或 generic submit-looking controls 混在一起，导致 scout 像是看到了可继续的 `wp_comment` submit surface；但 fresh full-page probe 之后，实际只剩静态文章、相关推荐、标签云、footer，以及一个站内搜索表单。遇到这类 surface 时，不要因为首轮 scout 命中了弱 submit candidates 就硬推进。应做一次 bounded 复核：至少滚完整页、确认 `form#commentform` / `form[action*="wp-comments-post"]` / `textarea[name="comment"]` / `input[name="author"]` / `input[name="email"]` / `input[name="url"]` 等真实 comment selectors 是否存在，并区分 search form vs comment form。若 fresh browser-use + Playwright 都确认只有 site-search form、没有 comment/reply editor、也没有 author website/url slot，应直接按 `unsupported_surface_no_direct_backlink_slot` 收口，而不是把普通文章详情页继续当作可评论 backlink surface。
- 新近实战补充：Prismo / fedibird 这类 story+comment surface 可能出现“公开故事页确有 comment thread 且公开评论正文可保留 outbound `<a href>`，但真正评论权限仍被 account-confirmation gate 卡住”的链路。可复核信号包括：① 故事页评论区直接显示 `コメントを投稿するには サインイン する必要があります`；② 公开 `/auth/sign_up` 暴露 username + email + password + password_confirmation 的正常注册表单；③ 提交后首页/登录页给出 `本人確認用のメールを送信しました`、`メールアドレスの本人確認が必要です`、`アカウントの有効化について数分以内にメールでご連絡します` 一类确认邮件提示；④ exact Gmail search 对该 alias 仍为空。遇到这组信号时，不要把它误判成 hard login wall、无 public signup、或 comment surface 本身不支持 backlink。更合适的 authoritative 收口是 `WAITING_EXTERNAL_EVENT + EMAIL_VERIFICATION_PENDING`，并把 resume trigger 写成“等确认邮件 / magic link 到达后回到 sign-in 和目标 story 继续 comment flow”。
- 同一类 Prismo / fedibird surface 还要区分“site 支持 comment backlink slot”与“当前账号尚未能发评论”两件事：公开 thread 里如果已能观测到其他评论正文带真实 outbound anchor（例如 `<a href="https://...">...</a>`），那就说明 surface 本身可承载 direct backlink，当前 blocker 只是 account confirmation / mailbox continuation，而不是 `unsupported_surface_no_direct_backlink_slot`。
- 新近实战补充：在 classic WordPress / article-comment surface 上，如果 fresh public thread pre-check 已经直接命中当前 promoted URL 的 live backlink（同一 target thread 中已存在可见 `<a href="https://promoted/...">...`，且 verifier 能确认 `verified_link_present`），不要机械继续再发一条 guest comment。此时更合适的 bounded operator动作是：记录现有 live comment/permalink、截图与 rel/anchor 证据，说明该 surface 已有 public backlink 落地，然后直接按 authoritative live-link success 收口，避免把同域重复 comment 提交成 duplicate/spam 噪声。尤其当 task 的业务目标是“该 target surface 上验证 live backlink 已存在”而不是“必须新增一条不同 comment”时，应优先把 live verification 当作完成信号，而不是把 comment form 的可见性自动等同于还需要提交一次。
- 新近实战补充：DLE / DataLife Engine 风格 comment surface（如 `form#dle-comments-form`）可能确实提供真实 public comment/backlink slot：公开线程里已有多个 outbound anchor，comment form 也真实包含 name/email/textarea/submit，以及可见图片验证码（常见 `engine/modules/antibot/antibot.php` + `sec_code` 输入框）。这类站点不要因为是资讯/图片站就先入为主判成“无 direct backlink slot”；应先核对现有 live comments 里是否已经存在外链落地，再决定是否值得进入 bounded 提交流程。
- 同一类 DLE image CAPTCHA 还要注意另一个坑：即便 runtime 已配置官方 CapSolver recognition 路径，`ImageToTextTask` 的 `common` 与 `number` 模式对站点自管验证码也可能连续返回低质量/错误解，最终服务端只回 `Проверка, что вы не робот, не была пройдена` 之类统一 anti-bot 失败页，而不会给更细的字段级报错。遇到这种情况时，不要把“有官方 solver key”误等同于“该 image CAPTCHA 可自动打通”；若 fresh bounded solver retries（例如 1 次 common + 少量 number/refresh 重试）仍稳定落到同一 robot-check failure，应把该站按 `captcha_blocked` / `captcha_or_human_verification_required` 收口，而不是无限刷新验证码继续空转。

状态判断：
- `awaiting moderation / comment submitted` 只代表进入站点队列，不代表成功
- 成功定义：public live comment 存在，且 link verification 命中目标链接
- live comment 已出现但目标链接缺失/被清洗，应记为 `COMMENT_PUBLISHED_NO_LINK`
- 实战补充：不要把“页面里能看到 `https://target` 这串文本”误判成 backlink 已落地；不少 wp_comment surface 会把 URL 原样显示成纯文本而不是 `<a href>`。authoritative verifier 必须确认 live thread 中确有可点击 outbound anchor 命中目标链接；若只有纯文本 URL、无 `<a>`、或 anchor 不再指向 promoted URL，也应记为 `COMMENT_PUBLISHED_NO_LINK`
- 对 `COMMENT_PUBLISHED_NO_LINK`，默认应直接按终态失败收口（skip / terminal audit），不要再做无人值守自动重试；页面已经公开但链接没落地时，重复提交通常只会制造噪声、重复评论或额外反垃圾命中
- Akismet / CleanTalk / duplicate comment / spam 检测等，应记为 `COMMENT_ANTI_SPAM_BLOCKED`

反垃圾经验索引：
- Akismet：常见于“looks like spam / duplicate comment detected”
- CleanTalk：常见于“anti-spam / cleantalk / forbidden / blocked”
- WordPress 原生 `wp-comments-post.php` 也可能直接回错误页而不是 moderation preview；若页面标题是类似“Fallo al enviar el comentario”，并出现西语文案 `Comentario duplicado: ¡parece que ya había sido enviado antes!`（或等价 duplicate-comment 提示），应视为 `COMMENT_ANTI_SPAM_BLOCKED` / policy-review 类阻塞，而不是 `COMMENT_MODERATION_PENDING`
- 实战补充：有些站会先返回 WordPress flood/rate-limit 文案（如 `You are posting comments too quickly. Slow down.`），随后同一 truthful payload 在冷却后又落到 duplicate-comment 错误。这个组合也应按 `COMMENT_ANTI_SPAM_BLOCKED` / policy-review 收口：说明站点端已经把该评论视为可疑或已存在，不应继续无人值守重复提交
- 实战里这种 duplicate / flood-control 组合错误后应再做一次 bounded public thread re-check：确认公开页仍只有 comment form、没有 live comment / promoted link，再停止无人值守重试，避免制造脏重复提交
- 这些经验应沉淀到 references，不应扩成共享 kernel 的站点级条件分支

优先沉淀到这里的内容：
- 通用 comment surface cue
- 逐字输入、提交前校验、审核态识别
- 反垃圾系统经验索引
- comment live verification 经验

不要放这里的内容：
- 单站点一次性 workaround
- 具体账号凭据
- 临时手工接管细节
