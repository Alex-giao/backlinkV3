# Runtime Failure Taxonomy for Web Backlinker V3 Operator

这份 reference 承接从 `SKILL.md` 外置出来的运行时异常 / case-driven heuristics。

使用原则：
- `SKILL.md` 只保留稳定护栏：策略哲学、最小完备工具集、必要事实、run contract。
- 本文件承接 runtime anomaly taxonomy、收口偏差、表单/验证码/弹窗/截图链路等 case-driven 经验。
- 遇到真实任务中的细分异常时，先用 `SKILL.md` 的证据驱动原则判断，再按本文件查找最接近的异常簇。
- 若后续新增经验只适用于个别站点或单次 incident，优先写 repo docs / playbook / execution_state / casebook，而不是继续加粗本文件。

## Migrated Runtime Facts / Case Taxonomy

下面这段内容原先位于 `SKILL.md` 的 `### Runtime facts`，现整体迁移到 reference，避免核心 skill 被 incident archive 挤厚。

- 浏览器端口只用 `http://127.0.0.1:9224`
- 每个任务必须使用 task-scoped browser-use session
- 单任务默认 timebox：10 分钟
- 默认：1 URL = 1 worker session
- 如果共享浏览器 / Playwright runtime 出现连接抖动、stale lease、会话污染，先做 runtime health-check 与清理，再决定是否重试；不要把运行时故障误判成站点失败
- 共享 CDP 里可能同时存在多个相同 URL 的 tab / page，且其中某些 page 可能处于 1x1 或极小 viewport 污染态；做证据确认时要先检查候选 page 的 viewport 健康度，再选择真正可见、尺寸正常的 page，不要默认信任第一个 URL 命中页
- 如果 browser-use 已连上目标 URL，但 `state` 超时、页面读不出来、或 viewport 掉到 1x1 / 极小值，优先把它当成 tab 污染而不是站点失败：先列 tab / page，必要时新建一个可见 tab，再把当前 task-scoped session 切到健康 tab 后继续取证
- 对这种 tiny-viewport 污染，优先用低风险证据链补位：当前 URL、viewport 指标、tab 列表、截图、必要时直接在共享 CDP 新建可见 tab；在拿到健康 tab 前，不要基于损坏 tab 的空白 `state` 结果做站点结论
- browser-use 的 task-scoped session 可能绑定到它首次看到的 CDP 配置；如果同名 session 报 `already running with different config`，不要把它误判成站点失败。先关闭该 task-scoped session，读取共享 CDP 的 `/json/version`，优先改用返回的 `webSocketDebuggerUrl` 重新连同一个 task-scoped session，再继续健康 tab 检查与取证
- 对很长的 task id，`task-{taskId}-{suffix}` 这种 session 名可能被 64 字符截断到几乎看不出 suffix，增加与旧会话/旧 tab 混淆的风险。实操时优先用“短 task 片段 + 明确时间戳”的唯一 task-scoped session 名；只要仍然是当前任务专属即可，不必执着于完整 task id 全量入名
- 即使 session 成功连上共享 CDP、viewport 也健康，首次落点仍可能是别的任务遗留 tab；因此开跑后先读一次当前 URL / title / tab 列表，若不在目标站点就立刻 `open` 目标 URL，再开始截图和状态判断，不要基于 unrelated tab 的健康状态直接下站点结论
- 即使 `eval`/URL 读取还能成功，browser-use `screenshot` 也可能因为会话仍挂在 tiny / 污染 tab 上而超时；遇到这种“能读页面但截图卡死”的分裂状态，先重新做健康 tab 选择，必要时通过共享 CDP `/json/new` 创建新的可见 tab，并优先切到最新创建、viewport 正常的目标 URL tab 后再截图
- 某些 fresh task-scoped browser-use 会话会出现另一种分裂状态：`open` 已成功、`eval` 能稳定读到 URL/title/body/viewport、`screenshot` 也能成功，但 `state` 单独持续超时。不要因为 `state` 超时就把任务误判成站点失败或 runtime 全坏；应把它视为 browser-use `state` 子链路失稳，继续用同一 task-scoped session 的 `eval` + `screenshot` + tab 列表补齐 authoritative evidence，并在 `operator-evidence` / `visual_verification` / trace 里明确记录“state timeout，但 eval+screenshot 成功”，然后正常 `task-record-agent-trace` + `task-finalize`
- 如果 fresh task-scoped browser-use `state` / URL / title 都稳定可读，但 `screenshot` 仍持续超时，不要因此放弃收口；这通常说明 browser-use 的截图链路本身失稳，而不一定是站点失败。此时应保留 task-scoped browser-use 作为主交互/trace 证据，同时改用共享 CDP 上的 Playwright 做 authoritative screenshot 补位（直接打开同一目标 URL、确认标题/正文/控件后截图），再把 browser-use state + Playwright 截图一起写进 `visual_verification`、`operator-evidence`、`task-record-agent-trace` 与 `task-finalize`
- 还有一种更隐蔽的变体：browser-use `screenshot` 命令本身返回 command failed / non-zero，但目标 PNG 实际已经落盘，且 `eval` 仍然稳定可读。遇到这种“报错但文件存在”的截图链路，不要机械把该截图当作完全失败；先检查目标截图文件是否真的生成，再结合 fresh Playwright screenshot、DOM/eval 与必要的视觉复核判断页面状态。若文件已存在且内容与其他证据一致，可继续把它作为辅助 artifact 写入 `operator-evidence` / `visual_verification` / trace；若文件不存在或内容损坏，再退回 Playwright screenshot 作为 authoritative 补位
- 如果 browser-use `state`/DOM 文本已经稳定给出强证据（例如明确的 parked-domain / expired-domain 文案），但 Playwright 补位截图却落成空白页、白屏或无正文，不要反过来用这张坏截图否定前面的强证据链。应把它视为截图链路/渲染瞬态异常，明确记录“Playwright 补位截图无效”，并继续以 fresh browser-use state + 现有 scout/历史健康截图作为 authoritative evidence 写回 `visual_verification` 与 handoff，而不是把任务重新降回 `VISUAL_VERIFICATION_REQUIRED`
- 某些 submit URL 会返回 `200 + 正常 title`，但真实页面只剩一个未渲染的 SPA 外壳：fresh screenshot 近乎纯黑/纯色空白，`document.body.innerText` 为空，`#root` child count 为 0，`body` 里几乎只有空 root 和脚本标签，没有任何 CTA / 表单 / 登录注册 / 错误文案。遇到这种“不是经典 404，但也没有任何可交互内容”的表面，不要因为状态码正常或标题看起来像主页就继续盲探；应把“空白 screenshot + 空 bodyText + 空 root + 0 forms/inputs”的证据链一起写入 `operator-evidence`、`visual_verification` 与 `proposed_outcome`，通常按 `RETRYABLE / STALE_SUBMIT_PATH` 收口，而不是误判成营销页、视觉待确认或站点成功可达
- 如果 repo 处于 operator-only mode，repo-native agent loop 可能被显式禁用；此时不要把 `runAgentDrivenBrowserUseLoop` / `run-next` 当 fallback，而应继续走外部 operator + browser-use + `task-record-agent-trace` + `task-finalize` 的主链
- 当最终结论主要来自截图/视觉判断（例如页面其实是现有 dashboard/profile，而不是 submit form）时，写回 `task-record-agent-trace` 的 envelope 必须带 `visual_verification` 摘要；否则 `task-finalize` 可能把任务再次打回 `VISUAL_VERIFICATION_REQUIRED`，即使你已经补了截图
- 如果 fresh task-scoped browser-use 会话 + screenshot 明确显示的是 Cloudflare / access denied / human verification block page，而不是提交表单，不要继续把它当“只是可达但语义不清”。应直接把该截图结论写进 `visual_verification`（通常可归到 `captcha_or_human_verification`），再走 `task-record-agent-trace` + `task-finalize`，让任务稳定收口到 captcha/policy 类终态，而不是反复停在视觉待确认
- 但若 scout / 旧 artifact 里看到的是 Cloudflare/403/interstitial，而 fresh task-scoped browser-use 会话在同一 tick 里已经稳定读到健康 viewport、真实 submit form、可填写字段或后续确认页，不要被旧 scout 结论绑死。应以 fresh 健康会话的 DOM + screenshot + 当前 URL 为 authoritative evidence，继续推进或按真实结果收口；旧 scout 的 Cloudflare 只能记为瞬时/前置拦截证据，不能反向覆盖 fresh 成功提交证据
- 当页面被 fresh screenshot + DOM/field 证据确认是可直接提交的真实 form，但资料包缺少该表单的必填字段时，不要为了“多推进一步”而硬填占位值；应正确收口到 `WAITING_MISSING_INPUT`，并在 handoff 的 `proposed_outcome.wait.missing_fields` 中明确列出缺的字段，同时保留 `visual_verification`，这样 finalize 才能把任务稳定停在缺失输入而不是再次回到视觉待确认
- 对多步 submit/register form，如果当前页只露出一部分字段、而哪些字段真的是硬必填仍不清楚，可以做一次 bounded 的“安全字段探测”：只填写资料包里已有且不会伪造业务事实的字段（例如站点、邮箱、分类、密码），然后点击下一步 / Review 观察站点是否返回明确的缺字段错误页。若返回 `Missing Fields` / `All Fields Are Required` 一类通用错误，就把“已填写的安全字段 + 仍缺的业务字段 + 错误页文案/截图”一起写入 `operator-evidence`、`visual_verification` 与 `proposed_outcome.wait.missing_fields`，并收口到 `WAITING_MISSING_INPUT`；不要因为想知道完整校验规则而继续捏造地址、电话、城市等业务信息。
- 某些站点的提交表单会用 `target=_blank`、新 tab 或 popup 打开结果页；此时 browser-use 当前 tab 可能仍停在原表单，甚至 title 看起来没有变化。提交动作后不要仅凭原 tab 的 `state`/title 判断失败；应立刻检查 tab 列表，定位新开的结果页并切过去，再基于该 tab 的 URL、正文、截图和可见成功/确认文案做结论
- 某些目录页会同时暴露 “Add company / Add another company” 入口与 pricing / upsell 文案。若你实际点进的是 public signup gate，并且注册提交后出现了明确的站点错误（例如页面内联 error、`?error=`/`?mismatch=` 参数、无对应验证邮件），不要因为同页仍能看到 pricing copy 就先入为主判成 paid listing；应把注册门槛 + 失败信号写进 `visual_verification` 与 `proposed_outcome`，并在 `task-finalize` 后复核最终状态，防止 finalize 过度收口为 `paid_listing`
- 某些目录页会在 public submit CTA 后先弹出 auth/register modal，宿主页本身又同时带着 `Pro` / `Pro Access` / pricing copy。若 fresh modal 证据已经明确进入 email-code register/login gate，并且请求验证码后只落到 `If you have an account...` / code-entry prompt、精确 mailbox search 仍无对应站点邮件，不要让 finalize 仅凭底层页面的 upsell 文案把结果覆盖成 `paid_listing`。应继续把 authoritative 结论写成 `register_gate` + “验证码邮件未到 / auth attempt 未确认”，并在最终汇报里同时说明 operator 实测结论与 repo 最终状态的偏差。
- 反过来，某些站点会先让你顺利完成 public signup，甚至已经落到 `confirm your email` / `we've sent an email` 页面，但该站真正的推广入口随后被 fresh DOM/截图确认只是 `Advertise` / `Start Advertising` / sponsored placements / media kit 一类付费广告计划，而不是免费工具/公司 submit form。遇到这种“注册成功 ≠ 免费提交可达”的分裂面，不要因为还在等验证邮件就先收口到 `WAITING_EXTERNAL_EVENT`。应继续补 authoritative monetization surface（如 `/advertise`、pricing/contact-sales 页面）的正文、CTA 与截图；若证据明确只有 banner / sponsored article / digest sponsorship / contact-sales inventory，则把“public signup succeeded + paid promotion surface only + exact mailbox search result（即便为空）”一起写进 `visual_verification` 与 `proposed_outcome`，优先按 `SKIPPED / paid_or_sponsored_listing` 收口。
- 某些公开 submit 路径会先落到 login gate，再暴露可自动注册的 public register gate；如果注册提交后出现了明确站点错误（如页面内联 `Server error.`）且用同一组新凭据立刻登录仍返回 `Authentication failed` 一类错误，同时 mailbox 里也没有对应验证邮件，不要把它误判成单纯 `login_required` 或 `WAITING_SITE_RESPONSE`。应把“公开注册可达但注册失败”的证据链（submit→login/register gate→register error→immediate login failure→no email）写进 `visual_verification` 与 `proposed_outcome`，优先按 `RETRYABLE / REGISTRATION_ATTEMPT_FAILED` 收口
- 某些公开 submit 流程会先露出真实提交表单，但点击继续或 `Register` 后并不会出现可自动填写的注册表单，而是直接跳到独立的 sign-in 页面，且页面上只有 `Continue with Google/GitHub/...` 一类第三方 OAuth 按钮，没有任何 email/password/username 输入框。遇到这种情况，不要因为文案里有 `Register` 就继续假设存在可自动注册路径。应做一次 bounded register probe，再用 fresh screenshot + DOM/input 计数确认“仅第三方 OAuth、零本地输入字段”，然后把“真实 submit form 已可达，但最终被 social-OAuth-only login gate 挡住”的证据链写进 `visual_verification` 与 `proposed_outcome`，按 `WAITING_MANUAL_AUTH / DIRECTORY_LOGIN_REQUIRED` 稳定收口
- 某些公开 signup form 会把邮箱 local-part 自动抄成 username；如果你用的是 `support+alias@...` 这类 plus alias，站点可能把 `+`、过长 local-part 或附加计数直接带进 username，随后返回 `invalid characters`、`less than N characters` 一类内联错误。不要把这类错误误判成站点关闭注册或整体注册失败。应先用 fresh DOM/内联错误确认真正阻塞的是 auto-generated username，再在不伪造业务身份的前提下手工改成一个有效短 username，必要时绕开有副作用的前端 click handler（例如直接 `requestSubmit()`）；若随后成功进入 logged-in submit surface 或看到明确 success/pending 文案，应按真实提交结果继续推进，不要停在 auth gate 结论。
- 某些 React/Next submit form 看起来只有一个简单 URL 输入框，但直接用 `eval` 改 DOM value、甚至 browser-use 的局部输入回显，并不一定会驱动前端真实表单状态；此时点击 submit 后页面可能报 `URL is required` 之类的假阴性校验。不要立刻把它收口成缺失输入。应优先用 authoritative 交互再验证一次（例如 Playwright `locator.fill()` + 真正 click/submit），看是否会进入真实的下一步（如 signup/register gate、确认页或后续表单），再决定是继续推进还是按缺字段/站点错误收口
- 某些 Google Forms / Material 风格表单也会出现类似分裂：browser-use `state` 能正确识别题目与必填星号，甚至 `input` 命令看起来已命中 shadow input，但值并不会稳定写回真实表单状态，导致提交后你误以为“所有字段都还是空的”或根本看不到真正缺的是哪一项。遇到这种情况，不要仅凭 browser-use 层的输入回显下结论；应改用 authoritative Playwright `fill()` / 真点击提交做一次 bounded 安全字段探测，并结合 hidden `entry.*` inputs、提交后 same-URL inline 校验文案（如 `此问题必须填写` / `This is a required question`）与 fresh screenshot，确认到底是哪一个 truthful 必填字段缺失。若安全字段已能写入而唯一剩余缺口是联系人全名之类未提供信息，应按 `WAITING_MISSING_INPUT` 收口，而不是误判成表单失效或运行时故障
- 对 Google Forms 还要记住另一条高价值证据链：如果首屏下拉/单选在 browser-use 与普通 Playwright 点击里都不稳定、而你又需要判断“还能不能继续填”或“到底缺哪些 truthful 必填字段”，不要卡死在表层交互上。优先读取页面里的 `FB_PUBLIC_LOAD_DATA_`（或等价的 form metadata）来枚举当前分支的必填题、选项和 entry id，再与 dossier 做 truthfulness 对照。若元数据已经明确暴露大量创始人身份、电话/邮箱、融资状态、竞争对手、未来计划、图片链接等资料缺口，就可以把 `fresh screenshot + browser-use state + form metadata` 一起写进 `operator-evidence` / `visual_verification` / `proposed_outcome.wait.missing_fields`，直接按 `WAITING_MISSING_INPUT` 稳定收口；不要为了证明每一道题都可点击而继续和不稳定的 Material 下拉控件缠斗
- 某些 Elementor / AJAX submit form 的 `button[type=submit]` 虽然存在于 DOM，但按钮文本可能是空的、可见框异常（例如宽度接近 0、点击命中框在视口外），导致 Playwright 普通 `click()` 报 `Element is outside of the viewport`，而页面本身并不是真失败。遇到这种“字段真实可填、确认文案也应是同页 inline、但 submit 按钮几何异常”的情况，不要草率收口为视觉模糊或站点故障；应优先保留 browser-use/截图/DOM 证据，然后用 authoritative 方式触发原生表单提交（例如对 `form.elementor-form` 执行 `requestSubmit()`，或等价的真实 form submit），再观察 `admin-ajax`/同页成功文案是否返回。若拿到 `Your submission was successful.`、`submission was successful`、`pending review` 等明确确认信号，应按成功提交处理并写回 `visual_verification` 与 `proposed_outcome`
- 有些 submit 后弹出的 signup/register gate 会在输入或勾选 terms 后重渲染，导致 browser-use 之前抓到的元素 index 失效并报 `Node with given id does not belong to the document`。遇到这种 index-stale / re-render 失效，不要把它误判成注册失败；先刷新证据（重新读 state / 当前 tab / screenshot），必要时切到 Playwright 做 authoritative bounded continuation，再根据 fresh 页面结果判断成功、失败还是等待邮件
- 另一类 index-stale 发生得更早：在超长 `wp_comment` / article 页面上，即使同一个 task-scoped browser-use session 仍然健康、`open`/`eval`/`state` 都成功，只要你先滚到 comment 区再尝试沿用上一轮 `state` 里的 input index，browser-use 也可能立刻返回 `Element index X not found - page may have changed`。实战里同一页面的 comment textarea / author / email / url / submit index 甚至会在一次 scroll 后整体重编号（例如从个位数跳到 7800+ / 11000+）。处理原则：不要把这种报错误判成站点表单消失或 runtime 坏掉；在最终滚动/定位到目标表单后必须重新跑一次 fresh `state`，再使用那一版 index。若第一条 input 仍然报这种 index-not-found，而 Playwright 同页 fresh DOM 已证明确有真实表单，就把 browser-use 视为“probe/定位成功但 index addressing 不稳定”，保留该 probe 的 `state`/screenshot 作为 evidence，然后切到 shared-CDP Playwright 做 authoritative bounded fill+submit，而不是继续硬撞旧 index 或宣告站点失败
- 某些多步 submit flow 会在前几步看起来一路正常：点击 Continue 后不断进入新的 step 页面、页面也持续显示真实字段和 CTA；但其中某一层的关键状态（例如 location step 的 city/region/country、或前面已填写的 title/contact/category）其实没有被服务端稳定 carry forward。典型信号包括：隐藏 `countryId/regionId/cityId` 只部分写入、下一步页面的隐藏 carry-forward 字段突然回空、而页面表面仍继续显示“Website URL (Optional)”这类可继续表单。遇到这种“流程仍在前进，但 payload 传递已失真”的状态，不要因为还看得到 Continue 就继续盲提交流程；应把它视为 submission integrity / state propagation failure，补留 fresh screenshot + hidden-field/DOM 证据，通常按 `RETRYABLE / OUTCOME_NOT_CONFIRMED` 收口，而不是冒险发布脏 listing
- 某些真实免费 submit/register form 也会同时带 reCAPTCHA 或 challenge iframe；如果页面文案明确写着 `100% free` / free listing，而 fresh screenshot + DOM/state 又明确显示 human verification，不能只留 `visual_verification` 后就交给 finalize 自行猜测。应在 handoff 里显式写 `proposed_outcome = SKIPPED/captcha_or_human_verification_required`，并把“这是免费表单、不是付费墙”的证据一起写进 `detail` / `visual_verification.summary`；否则 finalize 仍可能被页面上的营销/注册语气误带到 `paid_listing`
- 对 Cloudflare / challenge 小组件，不要只因 DOM 里出现 `cf-chl-widget` / challenge iframe 就直接判成 captcha 阻塞。先确认它当前是“待完成挑战”还是“已通过”状态：如果 fresh screenshot 明确显示绿色勾选、`Success/成功` 一类通过文案，且提交按钮可用，就应把它视为已满足前置验证，继续真实提交流程；只有当页面仍要求人工完成 challenge、按钮不可提交、或提交后明确报 verification/captcha 错误时，才按 `captcha_or_human_verification_required` 收口。
- 进一步地，若公开 submit form 的 DOM 里能看到 `cf-chl-widget` iframe 或隐藏的 `cf-turnstile-response` 字段，但 fresh screenshot 仍是正常可填写表单、没有可见 challenge 文案，且提交按钮可正常点击，不要把“有 Cloudflare 控件痕迹”本身当成阻塞结论。应把它当作前置校验可能已自动满足，先做一次 bounded 真实提交；如果随后出现明确 thank-you / submission received / pending review 等强确认，就按成功提交处理，并把“challenge 痕迹存在但未构成阻塞”的事实写入 evidence/visual_verification，而不是误收口为 captcha。
- 某些 Cloudflare Turnstile 注册页在初始 DOM 里只露出 `div.cf-turnstile` 和隐藏的 `cf-turnstile-response` input，fresh screenshot 上甚至暂时看不到明确 challenge iframe/checkbox；此时不要仅凭“控件还没渲染清楚”就继续无限试探，也不要仅凭隐藏字段就直接下 captcha 结论。优先做一次 bounded authoritative submit：如果提交后页面明确返回 `Please verify that you are not a robot.`、`verification required` 等机器人校验文案，就应把“真实注册表单 + Turnstile DOM 证据 + 提交后明确验证失败文案”一起写进 `visual_verification` 和 `proposed_outcome`，按 `SKIPPED / captcha_or_human_verification_required` 稳定收口。
- 另一种常见变体是站点在表单 DOM 里只留下自定义隐藏标记（例如 `captcha_type=slider`），真正的 challenge 要等你点提交后才弹出成可见滑块拼图 / slider dialog。即使 runtime 已配置 CapSolver key，也不要因为“环境里有 solver”就默认继续把这类 custom slider 当作已支持路径：若 fresh submit 后出现了可见滑块拼图（如 `向右滑动填充拼图`）而当前 runtime 又没有该 challenge 的明确官方 solve/apply 实现，应把“提交前仅有隐藏 slider 标记 + 提交后才出现可见 slider challenge”的证据链一起写进 `visual_verification`、`operator-evidence` 和 `proposed_outcome`，按 `SKIPPED / captcha_or_human_verification_required` 收口，而不是误报成普通 register gate、manual auth，或因为 `captcha_solver_attempt` 没识别出 supported descriptor 就降成 `OUTCOME_NOT_CONFIRMED`。
- 某些公开 register gate / signup form 会在 DOM 里带着 invisible reCAPTCHA / badge / `g-recaptcha-response` 隐藏字段，甚至页面底部还有“activities on hold / reviewed later / revamping”之类站点级公告。不要仅凭这些背景信号就把任务提前判成 captcha 阻塞、manual auth 或 policy 停机。先用 fresh screenshot + DOM 确认当前主表面是否仍是可真实填写的本地注册表单；如果真正阻塞点其实是缺少 truthful 必填资料（例如 community `Member Type`、`City`、真实 profile 字段），应把 invisible captcha/on-hold 公告记录进 evidence，但仍优先按 `WAITING_MISSING_INPUT` 收口，而不是让次级信号盖过主阻塞。
- 某些真实 submit form 会同时出现可见的 reCAPTCHA / verify frame 与一组尚未补齐的 truth-based 必填字段（例如 `City`、员工规模、创始故事、经验教训长文本）。这类场景不要因为页面上“确实看到了验证码”就直接把任务收口成 captcha 阻塞；先判断是否连表单本身都还无法 truthfully 填完。若当前 dossier 已足以证明页面是真实 submit form，但缺失字段本身已经构成更早、更硬的阻塞，应把 captcha 作为次级证据写入 `operator-evidence` / `visual_verification`，同时仍优先按 `WAITING_MISSING_INPUT` 收口，避免让后置验证码盖过前置资料缺口。
- 有些 submit 表单提交成功后不会跳转新 URL，而是在同一路径上先短暂显示 `Submitting...`、禁用按钮，再原位切换成 `Tool Submitted!` / `Thanks for submitting...` 这类 confirmation surface。遇到这种 same-URL success，不要因为 URL/title 没变就误判失败；应多等一拍，重新抓 fresh state/screenshot，若正文已切到明确确认文案，就按成功提交处理并收口到 `WAITING_SITE_RESPONSE`。
- 反过来，某些带 Google reCAPTCHA / JetFormBuilder 一类前置校验的真实免费 submit form，在 authoritative 点击 `Send` 后也可能仍停留在同一路径或只多一个查询参数（例如 `?status=captcha_failed`），页面主体继续显示原表单、没有 success/error toast，但 reCAPTCHA 仍可见、`g-recaptcha-response` / `_captcha_token` 仍为空。遇到这种“same-form + captcha_failed 参数/信号”的返回面，不要把它当成普通 `OUTCOME_NOT_CONFIRMED` 或继续盲重试；应把“真实 submit form 已可达 + truthful submit attempt 已发生 + 返回 `captcha_failed`/等价验证码失败信号 + post-submit screenshot 仍见 captcha 且无成功确认”的证据链写进 `operator-evidence`、`visual_verification` 与 `proposed_outcome`，优先按 `SKIPPED / captcha_or_human_verification_required` 收口。
- 如果 `task-finalize` 因本地凭据库环境缺失（例如 `BACKLINER_VAULT_KEY` 未设置）而失败，不要把它误判成站点失败；先判断这次 handoff 是否真的需要落库账号/凭据。若当前 tick 的目标只是保留推进证据并让任务正确收口，可去掉 trace payload 顶层 `account` 后重新 `task-record-agent-trace` + `task-finalize`，避免 finalize 因 vault 写入失败而中断
- `task-prepare` 返回的 `account_candidate` / `account_credentials` 可能只是“public signup draft”，并不代表这些凭据已经成功落库或后续一定还能从 account registry / credential vault 读回。若 prepare 明确是首次公开注册场景、repo 里也读不到现成账号记录，就把这些 draft 凭据视为候选草稿而非 authoritative secret：允许在真实注册页选择一组新的 unattended 安全密码继续完成注册，但要在最终 envelope 里写明实际使用的账号状态；若本轮又碰到缺少 `BACKLINER_VAULT_KEY` 无法落库，则至少保留注册成功/已验证/已提交的证据并按上一条去掉顶层 `account` 完成 finalize，不要因为 draft 凭据与最终实际密码不一致就把任务误判为站点失败
- 对 Angular/dashboard stepper 一类目录站，最后一步 `Submit` 不一定跳新 URL 或出现更强 thank-you 页面；有时 authoritative 信号只是一条官方 `dashboard/startups/update/<id>?user_id=...` 成功 POST，而 UI 仍停在 review step 文案里。若同时已确认 logo/截图等前置要求满足、页面无新的 validation/captcha/auth 错误、review copy 明确写会进入站点审核，就应把这类结果视为 `WAITING_SITE_RESPONSE / SITE_RESPONSE_PENDING`，而不是因为页面没跳转就误回退成失败或人工阻塞
- `task-finalize` 还可能在 shared CDP 上漂到 `chrome://omnibox-popup...` 或其他 unrelated tab，导致 `live_page_url` / `link_verification` 变成明显错误的伪阴性。遇到这种 finalization drift，先比对 handoff 里的 authoritative 当前页、network submit 证据与 focused screenshot；若 operator 现场证据更强，应优先把它当作 finalize/task-state mismatch 记录，并在需要时先显式打开目标页再重跑一次 `task-record-agent-trace` + `task-finalize`，不要让漂移 tab 反向覆盖 operator 结论
- 当怀疑有 captcha / iframe / overlay 但全页截图不够明确时，优先补 focused screenshot / DOM iframe 证据，而不是只靠全文截图做结论
- 如果 scout / DOM 已经给出了可交互 iframe 的 `frame_url`，并且宿主页只是 marketing landing / 介绍页，不要停留在宿主页截图上反复判断。应优先在同一个 task-scoped browser-use session 里直接 `open` 该 `frame_url`（或等价的嵌入表单 URL）来确认真实 submit surface，再基于该 surface 的字段、按钮、截图和缺失资料决定继续提交还是 `WAITING_MISSING_INPUT`
- 如果视觉模型暂时不可用（例如 `vision_analyze`/外部视觉额度 429）但你已经拿到 fresh screenshot、browser-use `state` 和 DOM 证据，不要因此停在 `VISUAL_VERIFICATION_REQUIRED`。应基于这些一手证据手工写出 `visual_verification.summary`，明确当前页属于 homepage / register_gate / submit_form / captcha 等哪一类，以及支撑该判断的 CTA、字段、错误文案或 modal 线索，再继续 `task-record-agent-trace` + `task-finalize`
- 对 shadow DOM / 组件化表单，不要只信普通 DOM scrape 得到的 `required=false` 或空 label：browser-use `state` 往往能直接暴露 `required=true`、真实 field name 与可交互控件；当 plain DOM 与 browser-use 观察不一致时，优先补 browser-use `state`、focused field/iframe screenshot，再决定是 `WAITING_MISSING_INPUT` 还是 captcha/policy 收口
- 某些 Google Forms / 自定义 checkbox 表面会出现另一种“可见但不吃普通 click”的情况：Playwright/坐标 click、`label.click()`、甚至 JS click 看起来都命中了，但 `aria-checked` 仍保持 `false`，提交后才暴露唯一剩余错误（如 `此问题必须填写`）。遇到这种情况，不要把它误判成资料缺失或站点故障；应把“首次 submit 后仅剩该 checkbox 验证错误”的证据记下，再用 authoritative 键盘交互补位（先 focus 对应 `role=checkbox`，再 `press('Space')` / 等价原生切换），确认 `aria-checked=true` 后立即重提交流程
- 如果 repo/runtime preflight 显示 browser-use CLI 正常，但 shell 里直接执行 `browser-use` 提示 command not found，不要立刻把它当运行时故障；先检查 host 上的常见安装位（尤其 `~/.browser-use-env/bin/browser-use`）或 repo 的 browser-use bin 解析逻辑，再继续 task-scoped session 调试
- 另一种更隐蔽的 browser-use runtime split 是：preflight / `guarded-drain-status` 仍把 `browser_use_cli=true` 判为健康，但真实 task-scoped `browser-use --cdp-url ... state/open/...` 在运行时会因为宿主代理环境缺少 `httpx[socks]` 依赖而直接报 `Using SOCKS proxy, but the 'socksio' package is not installed.`。遇到这种情况，不要把它误判成站点失败，也不要只因为 preflight 是绿的就继续硬撞 browser-use；应把它识别为 browser-use 子链路的 host-env 故障，记录原始报错，并立即切到 shared-CDP Playwright 做同一个 bounded continuation（fresh open、表单/线程取证、必要时 fill+submit、再 public re-check）。同时在 trace / operator-evidence 里明确区分：`browser_use_failure = host proxy socksio dependency`，而站点本身仍可能成功提交或已出现 live backlink。
- 某些 repo tick 环境里，`package.json` 脚本会先跑 `corepack pnpm build`，而 build 可能因为无关的 TS/test 漂移失败，即使 `dist/cli/index.js` 其实已经可用。遇到这种“脚本包装层失败、但 dist CLI 仍能运行”的情况，不要把它误判成站点或 runtime 故障；应继续在 repo root 用 `corepack pnpm exec node dist/cli/index.js <subcommand> ...` 直接执行 `guarded-drain-status`、`claim-next-task`、`task-prepare`、`task-record-agent-trace`、`task-finalize` 等 repo-native 子命令，完成这个 bounded tick 后再单独报告 build 漂移问题
- 对外部 backlinkhelper repo 的 bounded tick，不要机械套用 hermes-agent 主仓规则去 `source venv/bin/activate`：这个 repo 可能根本没有本地 `venv/`。正常 repo-native 命令应直接在 repo root 跑 `corepack pnpm ...`；如果需要临时做自定义证据采集/手工组装 envelope，优先用 `corepack pnpm exec node` 调用 repo 已编译好的 `dist/` 模块（如 browser-use helpers、data-store helpers），避免把“没有 venv”误判成 runtime 故障
- 对 pricing/marketing 页面里点击 CTA 才出现的 modal submit form，不要只看 finalize 阶段的 `body_excerpt` / `visible_surface_excerpt` 空白就回退成模糊页：browser-use `state` + fresh screenshot 往往能证明 modal 确实存在且字段可见。若资料包缺少该 modal 的必填字段，应带上 `visual_verification`、字段证据和 `WAITING_MISSING_INPUT` handoff 直接稳定收口，而不是再次打回视觉待确认
- 对 `wp_comment` 场景，如果通过 `wp-comments-post.php` 直提后拿到了 `?unapproved=<id>&moderation-hash=...#comment-...` 这类 preview URL，并且页面已明确显示类似“评论正在等待审核/approval”的文案，不要再要求 live backlink 当场成立；这应先按 `WAITING_SITE_RESPONSE / COMMENT_MODERATION_PENDING` 收口。实战中 `task-finalize` 重新接共享 Chrome 时，可能漂到无关 top-chrome/omnibox 页面，导致 `current_url=chrome://omnibox-popup...`、`link_verification=link_missing` 之类伪阴性。此时应以提交阶段拿到的 preview 证据和 moderation 文案为 authoritative pending evidence，把这类 finalize 漂移视为当前 finalization 取证缺陷，而不是评论失败或 live-page 真缺链。
- 某些站点首页的 "Get Listed" / "Add Your Company" 弹窗表面上是可填写表单，但文案会明确写出 `pricing options`、`contact us today`、赞助/销售咨询等信号；这类页面不要因为看到了必填字段就误判成可直接免费提交或 `WAITING_MISSING_INPUT`。应把弹窗截图、可见文案和字段证据一起写入 `visual_verification`，并优先按 `paid_listing` / `paid_or_sponsored_listing` 收口
- 某些 submit 页表面上是可直接提交的真实 form，但免费审核会明确附带 reciprocal backlink / footer link 条件（例如 `Free Pack $0` 同屏写着必须把对方 partner link 放到你站点 footer，甚至直接给出 HTML snippet）。这类情况不要因为页面可达、也有 Website 输入框和 Submit 按钮，就误判成还能继续自动提交；它本质上是需要外部站点改动/策略批准的 policy boundary。应把“真实 submit form + 明确 reciprocal backlink 条件 + 站点要求的具体链接/snippet”一起写进 `operator-evidence`、`visual_verification` 与 `proposed_outcome`，优先按 `WAITING_POLICY_DECISION / RECIPROCAL_BACKLINK_REQUIRED` 收口，而不是误收口成普通 paid listing、`WAITING_MISSING_INPUT` 或继续硬提交流程
- 但若 reciprocal backlink / paid priority / advertising upsell 文案只是在提交成功后的 confirmation surface 上出现，同时 fresh URL、标题、正文或截图已经明确给出 `Submission Received`、`thank you`、`review queue`、`pending review` 等强确认信号，不要反过来把这类 post-submit upsell 误判成新的前置 policy blocker。应把 authoritative 结论仍写成成功提交（通常 `WAITING_SITE_RESPONSE`），并在 `detail` / `visual_verification.summary` 里明确说明“reciprocal backlink / paid review 只是提交后的可选加速项，不影响本次 submission accepted 结论”。
- 某些真实 submit form 在 authoritative 提交后会返回 `200`，但页面既不给 success/thank-you/pending-review，也不给 inline error；它只是把同一张表单重新渲染成空白初始态，原先填写的值全部清空，URL 可能只多一个 `#` 或几乎不变。遇到这种“POST 确实发出去了，但响应只回 blank form”的静默结果，不要把它误判成成功提交，也不要仅凭 200 就收口到 `WAITING_SITE_RESPONSE`。应尽量保留 request payload 已发送的证据（network/Playwright POST、提交前后截图、表单值被清空、无 success/error 文案），并优先按 `RETRYABLE / OUTCOME_NOT_CONFIRMED` 收口。
- 某些公开 submit/add form 确实是真实表单，但提交对象被强约束为第三方 marketplace / app-store listing URL（例如页面只接受 Google Play、App Store、Steam、GOG、Meta、Nintendo、PlayStation、Xbox 等商店详情链接），而不是普通 SaaS 官网。遇到这种“表单真实可达但对象类型不匹配”的场景，不要把 promoted site 的官网 URL 硬塞进去制造脏提交。应把“真实表单 + 必填 URL 类型约束 + 页面示例链接/支持平台 + 当前资料包只有官网、没有受支持 store URL”的证据链写进 `operator-evidence`、`visual_verification` 与 `proposed_outcome`，优先按 `WAITING_MISSING_INPUT` 收口，并在 `missing_fields` 中明确要求 `supported_app_store_listing_url`；如果 campaign 明确不做这类 app-only 目标，再由人工决定 skip/remove，而不是由 operator 擅自乱填官网
- `task-finalize` 的 Playwright 收口即使拿到了明确 handoff，也仍可能因为共享 CDP tab 污染而在 `finalization.json` 里落到 unrelated page 的 `current_url` / `body_excerpt`。如果你已经在 handoff 中明确写入了 `proposed_outcome`、`visual_verification`、`operator-evidence` 与正确的 `agent_trace_ref`，最终任务状态通常仍能稳定收口；因此 finalize 后要优先复核任务最终 `status`/`wait`/`execution_state` 是否正确，不要仅因 `finalization.json` 截到别站页面就误判这次 operator tick 失败
- 对 `wp_comment` 还要警惕另一种 finalize 漂移：目标文章正文里常有外链/作者 CTA，finalization fresh page 可能被共享 CDP 导航到其中一个 external link（例如 Telegram 联系页），导致 `link_verification.live_page_url` 变成 unrelated external page、`link_missing` 也随之失真。若 operator tick 里已经拿到了 duplicate-comment / moderation-preview / public-thread re-check 这类更强的一手证据，应以这些 task-scoped submit artifacts 为 authoritative truth，把 external-page finalization drift 视为收口取证噪声，而不是把任务结论改写成“目标页外链不存在”
- 如果 `task-finalize` 第一次因为 shared CDP page 选择漂到 unrelated page 而写出了错误的 `live_page_url` / `link_verification`，但 task-scoped evidence 已经强到足以确认真实目标页，应优先做一次 bounded repair：先在共享 CDP 里显式打开目标 `current_url` / `target_url`，确保浏览器里存在该 exact URL 的健康 tab，再用同一个 envelope 重新执行 `task-record-agent-trace` + `task-finalize` 覆盖错误持久化状态；不要直接手改 task JSON，也不要把第一次 finalize 的伪阴性当成 authoritative 站点失败。
- 新近实战补充：即使共享 CDP 里其实已经存在目标 exact URL 的 tab，`task-finalize` 仍可能因为 `withConnectedPage()` 默认挑选“最后一个非空页”而继续漂到别的任务遗留页（实测从 `www.codeforphilly.org` 漂到 `germanistika.unizd.hr`）。这说明“`/json/list` 里已经有 exact target URL”本身不保证第一次 finalize 会选中它。遇到这种情况，仍按上条 repair 做：先在 shared CDP 里再显式 `goto/open` 一次目标 exact URL，让它成为最新健康页，然后重新 `task-record-agent-trace` + `task-finalize`。复核时以第二次 finalize 后的 authoritative task JSON 为准；期间第一次 finalize 留下的 mismatch blocker/evidence 可能仍保存在 `execution_state` 中但会标成 `resolved`，不要把这类历史噪声误报成最终未修复 blocker。
- 某些 WordPress comment surfaces 在 truthful submit 后不会跳到 moderation preview，而是直接把请求落到 `wp-comments-post.php` 错误页；如果标题类似 `Fallo al enviar el comentario`，正文出现 `Comentario duplicado: ¡parece que ya había sido enviado antes!` 或等价 duplicate-comment 文案，这通常说明触发了 duplicate/anti-spam barrier。正确做法是：保留该错误页 screenshot + body text，再回公开文章页做一次 bounded re-check；若公开页仍只有 comment form 且没有 live Bell Borer / promoted-link comment，就优先按 `WAITING_POLICY_DECISION / COMMENT_ANTI_SPAM_BLOCKED` 收口，避免继续无人值守重复提交
- 对“public signup/register gate 明确可达，且提交后出现站点内联错误”的场景，哪怕你已经补了 focused screenshot、`visual_verification.classification=register_gate` 和明确的 `proposed_outcome = RETRYABLE / REGISTRATION_ATTEMPT_FAILED`，`task-finalize` 仍可能因为它自己的 finalization page assessment 只看到低置信 auth surface，而把最终 wait 收口成 `VISUAL_VERIFICATION_REQUIRED`。遇到这种情况，不要误以为 operator 没留够证据；应先确认 trace / screenshot / probe / visual_verification 都已写入 repo，再以“repo 最终状态为准”汇报，并在复盘时把它识别为 finalize 侧的 guard/分类偏保守，而不是站点结论缺失
- 还有一类更具体的验证码变体：`task-finalize` 可能已经真的执行了官方 CapSolver recognition（例如 `captcha_solver_attempt.solved=true`、`captcha_kind=image_to_text`、并点击了 submit），目标站随后也明确返回了内联错误，如 `验证码不正确` / captcha incorrect。即便如此，final outcome 仍可能异常地写成 `VISUAL_VERIFICATION_REQUIRED`，错误文案甚至沿用 `no visual verification payload was provided`，而实际上 envelope 里已有 `handoff.visual_verification`、task JSON 里也会显示 `visual_gate_used=true`。遇到这种组合时，要把它识别为 finalize/runtime persistence bug，而不是 operator 漏传视觉摘要：最终汇报时应同时说明“CapSolver 已尝试但验证码被站点判错”的真实 blocker，以及“repo authoritative state 目前仍落成 RETRYABLE + VISUAL_VERIFICATION_REQUIRED”的持久化偏差。
- 对某些首页 auth modal + dropdown submit 站（例如先 `Add your > Company` 再进入 auth chooser 的目录站），即使你已经补了 fresh browser-use / Playwright 证据，并且 handoff 明确写了“email 路径是现有账号登录、另一路只剩社交 OAuth、没有公开 signup / submit form”，`task-finalize` 仍可能因为 finalization 自己抓到的是主页 join 文案或低置信 auth surface，而把结果重新抬回 `RETRYABLE / SIGNUP_FLOW_AVAILABLE`。遇到这种情况，要把它识别为 finalize 侧对 signup continuation 的过度乐观，而不是 operator 没探够：保留 `visual_verification`、focused auth screenshot、`Continue another way` 分支证据和明确的 `proposed_outcome`，并在最终汇报里同时说明“repo 最终状态”和“operator 实测结论”之间的偏差，方便后续 guard/分类器修正
- 某些 submit URL 表面上不是付费提交页，而只是普通目录 listing / home feed；页面卡片里会混入大量 `Paid` / `Freemium` / `Free Trial` 标签，但 fresh screenshot + DOM 已确认当前 surface 根本没有 submit/add/login/register CTA，且 `form`/可见输入都是 0。遇到这种“generic listing feed 被卡片价格标签污染”的返回面，不要让 `task-finalize` 仅凭正文里的 `Paid` 词样把结果覆盖成 `paid_listing`。应把 authoritative 结论继续写成 `404_or_stale_submit_path` + `RETRYABLE / STALE_SUBMIT_PATH`，并在最终汇报里同时说明 repo 最终状态与 operator 实测结论的偏差。
- 某些站点会把关键 continuation CTA 渲染成“看起来像按钮”的 `<a class=\"button\">...</a>`，而不是原生 `<button>` / `<input type=submit>`。这类页面上，通用 click helper 若只扫 button/submit，容易停在资格检查页或 pricing/landing step，误判成 `OUTCOME_NOT_CONFIRMED`。遇到这种表面（典型信号：页面正文明确写着 proceed / continue / submit，但自动化日志里 `button not visible` 或没有命中任何可点击控件），要补一轮 authoritative DOM 枚举：同时检查 `a, button, input[type=submit], input[type=button]`，必要时直接跟进可见 anchor 的 `href`。若 anchor 继续后露出了真实表单，应以这个 deeper form 为 authoritative submit surface，不要把前一步的“假按钮页”当最终结论。
- 某些旧式 start page / intranet links page 会制造另一种 scout 假阳性：页面正文里充满 `Katalog`、`submit`、`website`、`email` 等词，甚至还挂着一个看起来像按钮的站内登录控件或隐藏 input，导致 scout/fragment 误以为这里存在 submit surface。若 fresh screenshot 实际显示的是网址导航/工具聚合页，而唯一显眼 candidate 继续后又落到 auth-gated 内网目录、实验目录或其他非公开工具页，应立刻做一次 bounded candidate reprobe（直接打开最像 submit 入口的 href），再以该 reprobe 的真实可达性作准：若 candidate 明确是内网/认证页或与 backlink flow 无关，就应按 `SKIPPED / unsupported_surface_no_direct_backlink_slot` 收口，不要继续停在 `RETRYABLE / OUTCOME_NOT_CONFIRMED`。
- 还有一种相近但更隐蔽的 scout 假阳性：原始目标 URL 会 30x/JS/站点级重定向到一个完全无关的公共首页或资源页（实战例：声称是 `wp_comment` 文章页，fresh browser-use + shared-CDP Playwright 却稳定落到另一个 hostname 的 government/resource homepage）。这类页面往往正文里仍有大量导航链接、搜索表单或 `name` / `website` 词样，容易让旧 scout fragment 继续误报 `submit_surface` / `form_surface`。遇到这种“claimed target -> unrelated homepage”分裂时，不要沿用 scout 的 submit 候选继续盲点；应立即以 fresh 当前 URL、最终 hostname、表单 action、是否存在 `textarea` / `wp-comments-post.php` / reply CTA、以及视觉截图为 authoritative truth。若 fresh 证据表明这里只是无关 homepage/resource hub、没有 comment 或提交入口，就应直接按 `SKIPPED / unsupported_surface_no_direct_backlink_slot` 收口，并在 detail 中明确写出“claimed URL 已重定向到 unrelated surface”，避免把导航站/resource hub 误当成可推进 submit flow。
- 另一类需要单独记住的 scout 假阳性是 third-party status/outage/report 页面：正文里会出现 `Check website`、`Report an issue`、`Forum & Comments`、rating stars、`I have a problem with <other domain>` 等 CTA，看起来像“可互动 surface”，但它的业务语义仍然只是围绕别人的站点故障/评论讨论（实战例：UpdownRadar 的 `markdowntoword.io` status page）。遇到这种页面，不要因为存在 report/comment UI 就把它当成可投稿 directory / comment surface；应先核对页面主语是不是目标推广站、是否存在真实 submit-product/profile/article flow、以及是否有可 truthfully 放入 promoted URL 的 direct backlink slot。若 fresh browser-use + Playwright + visual evidence 都表明这里只是第三方 outage/report discussion，没有 Exact Statement 这类 promoted site 的 listing/profile/website 提交位，就应直接按 `SKIPPED / unsupported_surface_no_direct_backlink_slot` 收口，而不是继续尝试把 promoted site 塞进不相关的故障讨论。
- 不要把 `execution_state` 当成精确工时统计器。它本质上是 repo 状态层的执行记忆摘要（frontier / blockers / evidence / discovered_actions / reusable_fragments），其中 evidence 的 `created_at` 是写回状态层时刻，不是浏览器内每个真实页面事件的原始时间线；而且 artifact 路径如 `taskId-agent-loop.json` / `taskId-scout.json` 是固定文件名，重复运行会覆盖同名 artifact，`execution_state` 内部很多条目也按稳定 id upsert。结论：`execution_state` 适合判断“当前走到哪、看到了什么、卡在哪、最近一轮怎么收口”，不适合直接拿来计算任务从入队到收口的累计纯执行时长；做时长分析时，要把队列等待、cron 节拍、重试回合、外部等待（邮件/目录审核）与最后一轮活跃执行分开估算
